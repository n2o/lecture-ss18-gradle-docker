Allgemein: PATH

Suchpfad, den Ihr Betriebssystem zum Nachschlagen verwendet (z.B. für Binaries) 

λ echo $PATH
/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin

CLASSPATH von Java

Suchpfad, an denen der Java-Compiler u.A.
nach Class- / Jar-Dateien sucht

Standard: .
(das aktuelle Verzeichnis)

Das funktioniert auch erstmal ganz gut so...

λ javac HelloWorld.java
λ java HelloWorld
Hello, world!

... aber hat man externe Bibliotheken (z.B. JUnit fürs Testing), dann muss ein anderer Classpath gesetzt werden 

λ javac -cp .:junit-4.12.jar:hamcrest-core-1.3.jar MyLibTest.java

λ java -cp .:junit-4.12.jar:hamcrest-core-1.3.jar \
       org.junit.runner.JUnitCore MyLibTest
JUnit version 4.12
.
Time: 0,004

OK (1 test)

Neues Projekt im Ordner "myproject" anlegen

λ nano build.gradle

Plugin für gewünschte IDE hinzufügen

plugins {
    id 'java'
    id 'application'
    id 'idea'          // für IntelliJ
    id 'eclipse'       // oder für Eclipse
    // ...
}

Projektdateien für IDE anlegen (hier am Beispiel von IntelliJ) 

λ gradle idea

Projekt in IDE öffnen und bearbeiten

λ intellij .

Projekt ausführen

λ gradle run

Projekt testen

λ gradle test

Externe Abhängigkeit fürs Testing einfügen (werden mit "testCompile" markiert) 

dependencies {
    testCompile 'junit:junit:4.12'  // nur in Tests eingebunden
}

Externe Abhängigkeit für Produktivcode einfügen (werden mit "compile" markiert) 

dependencies {
    testCompile 'junit:junit:4.12'
    compile 'com.google.guava:guava:24.1-jre'  // <--
}

Container

Instanz eines Images.

Image

Fertige Umgebung, in der Code ausgeführt werden kann. Besteht aus (mehreren) Layern.

Layer

Eine Instruktion zur Erstellung eines Images.

Leichtgewichtiger als Virtuelle Maschinen, da u.A. der Kernel des Hosts mit Docker geteilt wird

Nur die notwendigen Programme werden in einem Docker-Image mitgeliefert

Ein Docker-Container lässt sich innerhalb kürzester Zeit erstellen — und auch genauso schnell wieder wegwerfen

Vorteile

  • Portabilität von Anwendungen
    • Trennung von Anwendung und Infrastruktur
    • weniger Abhängigkeiten
  • Einfacher
    • man muss kein Linux-Profi sein, um ein Programm (bspw. einen Webserver) zu installieren
  • Weniger Sicherheitslücken
    • Minimale Images haben nur die notwendige Software
    • ... aber Kernel wird geteilt

Gleiche Umgebung
für Development, Testing und Production

macOS

Windows

Win 10 Pro, Enterprise und Education.

Datenbanken

Möchten wir Daten speichern, bieten sich Datenbanken an. Hier ein Beispiel mit dem Key-Value-Store Redis (ähnlich einer HashMap).

λ docker run -p 6379:6379 redis:alpine

Um sich mit der Datenbank zu verbinden,
verwenden wir hier Jedis

compile "redis.clients:jedis:2.9.0"

Datenbanken

Zählt Programmstarts und speichert
den Wert in der Datenbank

import redis.clients.jedis.Jedis;

public class App {
    private static void init(Jedis jedis) {
        if (jedis.get("starts") == null) jedis.set("starts", "1");
    }
    public static void main(String[] args) {
        Jedis jedis = new Jedis("localhost");
        init(jedis);
        System.out.println("Program starts: " + jedis.get("starts"));
        jedis.incr("starts");
    }
}

Dynamisch Code aktualisieren

Möchte ich den Code in einem Container während der Laufzeit ändern, muss ich ein Volume Binding anwenden

Dafür erstellen wir eine minimale Webseite

<!-- Datei: index.html -->
<html>
<head><title>ProPra1</title></head>
<body>
  <h1>Vorlesung zu Gradle und Docker</h1>
</body>
</html>

Dynamisch Code aktualisieren

Dann starten wir einen nginx-Webserver und laden die Datei index.html dort hinein

λ docker run -p 80:80 -v `pwd`:/usr/share/nginx/html nginx:alpine

Der Parameter `-v` sorgt dafür, dass alle angegebenen Dateien bidirektional gelesen und geschrieben werden

Im Browser können wir nun http://localhost aufrufen

Wenn wir nun index.html verändern, ändert sich auch der Inhalt des Containers und damit auch die Webseite
